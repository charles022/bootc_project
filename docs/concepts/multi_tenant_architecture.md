# Multi-tenant architecture

## What

A layer on top of the bootc host that turns the workstation into a small **multi-tenant control plane**. The host image stays minimal and trusted; each human tenant is represented on the host only by a non-login service account that owns a rootless Podman runtime, a tenant storage namespace, a credential namespace, and a set of host-managed systemd/Quadlet units. Human users never log into the host — they reach their environment through container-level entry points (SSH into a dev container, an agent web UI, a messaging bridge) exposed via host-managed Cloudflare Tunnel sidecars.

This is an **additive** layer. The existing single-machine `devpod` (see `concepts/ownership_model.md`, `reference/quadlets.md`) continues to be the local GPU dev pod for the workstation owner. The multi-tenant layer adds tenant-isolated pods alongside it, generated by `platformctl` (see `reference/platformctl.md`).

## Why

The single-machine model in `concepts/ownership_model.md` already separates *machine* concerns from *workload* concerns. As soon as more than one human (or more than one autonomous agent) shares the box, that split is no longer enough: workloads need to be isolated from each other, not just from the machine. The multi-tenant layer extends the same ownership rule one level down — the **host** owns everything platform-wide, each **tenant** owns its own slice of compute/storage/credentials, and **agents** inside a tenant slice can be powerful within that slice without being able to administer the host or peer at other tenants.

The alternatives we rejected:

- **One Linux user per human, with login + sudo.** Gives guests a real shell on the host; every guest package install, agent crash, or compromised credential becomes a host concern. Violates "the host is not a workstation for guests."
- **One giant shared dev container.** Co-tenants share filesystem, env, sockets, and process namespace. No isolation, no per-tenant quota or credentials.
- **A full hypervisor / Kubernetes cluster.** Too heavy for a single workstation; loses the bootc/ostree update story.

Rootless Podman gives us "root inside container ≠ root on host" cheaply, and Quadlet gives the host a declarative way to manage tenant pod lifecycles through systemd. That combination is the minimum viable isolation layer that fits the bootc model.

## Implications

### High-level shape

```text
Fedora bootc host (control plane, minimal, immutable)
├── admin login (real human, the only host login)
├── platformctl                   admin CLI: tenant CRUD, lifecycle
├── agentctl host endpoint        agent-facing CLI (planned)
├── openclaw-broker               credential broker (stub today; planned)
├── tenant-onboard pod template   Quadlet template, rendered per-tenant
└── per-tenant rootless Podman runtimes
      ├── tenant_alice (non-login service account)
      │     └── alice-onboard.pod
      │           ├── alice-cloudflared
      │           ├── alice-onboard-env
      │           ├── alice-openclaw-runtime
      │           └── alice-credential-proxy
      └── tenant_bob (non-login service account)
            └── ...
```

### The host is the trusted control plane

The host image owns:

- tenant creation, deletion, disable/enable
- tenant rootless Podman ownership (subuid/subgid allocation, storage roots)
- systemd / Quadlet lifecycle for tenant pods
- cloudflared configuration and tunnel tokens
- credential storage, encryption, scoped grants (planned)
- agent / environment provisioning (planned)
- updates and rollback through bootc/ostree
- storage allocation and quotas (planned)
- policy enforcement (planned)

The host image must not accumulate per-tenant tooling, per-tenant Python environments, per-tenant SSH keys, or per-tenant project files. Anything tenant-specific lives under `/var/lib/openclaw-platform/tenants/<tenant>/` (see `concepts/tenant_storage_layout.md`).

### Tenants are platform identities, not host logins

A tenant is not a person who can `ssh root@host`. A tenant is a platform identity composed of:

- a non-login host service account, e.g. `tenant_alice`
- a tenant policy file (planned)
- a tenant credential namespace (planned)
- a tenant storage namespace
- a tenant rootless Podman store
- tenant-scoped systemd/Quadlet units under `/etc/containers/systemd/users/<UID>/`
- tenant-scoped cloudflared routes/tokens (planned)
- tenant OpenClaw agent objects (planned)

See `concepts/tenant_identity_model.md` for the identity, subuid/subgid, and rootless-Podman details.

### Agents are powerful inside their slice, weak across the host

Agents — LLM-driven processes inside `openclaw-runtime` containers — may have full sudo inside their dev environment container, install packages, run compilers, edit code. They cannot:

- `ssh` to the host
- `sudo` on the host
- call `podman`, `systemctl`, `cloudflared`, `mount` directly on the host
- read other tenants' volumes, secrets, or pods
- read the host credential broker database directly

Agents request platform actions through a constrained CLI/API (`agentctl`, planned — see `reference/agentctl.md`). Every request goes through tenant identity, policy, quota, credential-grant, and storage-grant validation before the host generates a Quadlet and starts a rootless container.

The guiding rule is:

> Maximum capability inside the assigned tenant environment.
> Minimum authority across the host boundary.

### Pod composition

Each tenant typically runs one or more pods composed of:

| Container | Role | Trust |
|---|---|---|
| `cloudflared` sidecar | Public ingress via Cloudflare Tunnel | platform-controlled, immutable |
| `dev-env` | Tenant's mutable shell / toolchain / project files | semi-trusted, sudo-inside-container |
| `openclaw-runtime` | Agent control loop, tool dispatch | semi-trusted, locked down |
| `credential-proxy` | Local socket for scoped credential access | platform-controlled, immutable |
| `messaging-bridge` (optional) | Signal / WhatsApp / email integration (planned) | semi-trusted |

Rules for what may share a pod are in `concepts/tenant_identity_model.md` § "Pod boundary".

### What is built today vs. planned

**Built today (Phase 0 + Phase 1 + Phase 2 + Phase 3):**
- `platformctl tenant create | list | inspect | disable | enable | delete | verify-isolation` — admin CLI on the host
- `platformctl tunnel set-config | set-credentials | show | list` — per-tenant cloudflared config / credentials install (root-owned)
- `platformctl credential add | list | delete | rotate` + `platformctl grant add | remove | list` + `platformctl audit tail` — Phase-2 broker admin CLI (`concepts/credential_broker.md`)
- `platformctl agent create | list | inspect | start | stop | delete` + `platformctl policy show` — Phase-3 provisioner admin CLI (`concepts/agent_provisioning.md`)
- `agentctl` tenant-side CLI inside the openclaw-runtime container with `create-agent | list-agents | inspect-agent | start-agent | stop-agent | delete-agent | policy-show | ping`
- per-tenant non-login service account with collision-free subuid/subgid allocation
- per-tenant `/var/lib/openclaw-platform/tenants/<tenant>/` storage layout (now including a real `policy/policy.yaml` and `agents/<agent>.json` records)
- per-tenant Quadlets rendered into `/etc/containers/systemd/users/<UID>/` for both the onboarding pod and any agent pods
- onboarding-pod template (cloudflared sidecar + onboarding-env + openclaw-runtime + credential-proxy)
- real **`openclaw-broker`** daemon with Fernet-encrypted credential store, grant table, append-only audit log, admin socket (UID 0 via `SO_PEERCRED`), and per-tenant sockets chowned to `tenant_<tenant>`
- real **`credential-proxy`** container that bind-mounts its tenant's broker socket and re-exposes a pod-local agent socket; refuses admin verbs at the pod boundary
- real **`openclaw-provisioner`** daemon with per-tenant policy + quota engine, broker-cross-checked credential validation, agent Quadlet rendering, systemd start/stop integration, append-only audit log
- **`openclaw-runtime`** image now ships `python3` + `agentctl` and bind-mounts the per-tenant provisioner socket for self-provisioning
- multi-tenant directory layout under `/var/lib/openclaw-platform/`

**Planned:**
- messaging bridges (Signal / WhatsApp / email) — Phase 4
- backup, restore, audit-log retention — Phase 5
- cloudflared route automation via the Cloudflare API (Phase 1 ships *config installation*; *tunnel creation* still requires an external `cloudflared tunnel create` call)
Interactive `openclaw-onboard` flow inside the onboarding-env container — Phase 4
- OAuth / login-URL flow — Phase 4
- sealed / HSM-backed broker master key, scheduled credential rotation — Phase 5 hardening
- `agentctl create-env`, `attach-storage` / `detach-storage`, per-agent CPU / memory cgroup enforcement — Phase 4+

The phasing follows the original Multi-Tenant Architecture proposal (§20 of that document). See `design/multi_tenant_architecture.md` for the full 23-section specification and `roadmap.md` for the live status.

## Non-negotiable design rules

These are the same rules as in the proposal document, restated here so that any future PR can be checked against them:

**Host rules**

1. Only the admin logs into the host.
2. No guest SSH to the host.
3. No guest `sudo` on the host.
4. No guest packages installed on the host.
5. No guest project files on the host except inside controlled tenant volumes.
6. No agent access to the host Podman socket, host systemd, host cloudflared API/token, or raw credential database.

**Tenant rules**

1. One non-login service account per tenant.
2. Rootless Podman under that service account.
3. Separate subuid/subgid range per tenant.
4. Separate storage namespace per tenant.
5. Separate credential namespace per tenant.
6. Separate cloudflared route/token per tenant or pod.

**Agent rules**

1. Agents request actions; they do not administer the host.
2. Agents use `agentctl`; they do not run raw `podman` / `systemctl` / `cloudflared`.
3. Agents receive scoped credentials; they do not own master credentials.
4. Agents use selected volumes; they do not choose arbitrary host mounts.
5. Agents run rootless; no privileged containers.

**Environment rules**

1. Dev containers may allow internal `sudo`.
2. The OpenClaw runtime should be more locked down than the dev container.
3. The cloudflared sidecar is platform-controlled.
4. The credential proxy is platform-controlled.
